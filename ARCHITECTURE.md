## Design principles

- **No info held on a server.** No accounts, no message/chat history persistence,
  no persisted room list. Sessions are in-memory only and disappear the moment the
  last peer leaves (`SessionManager.js`). Media is P2P wherever possible; even when
  a TURN relay is needed as fallback, it's blind-relaying encrypted bytes, not
  storing or inspecting anything.
- When weighing a new feature, check it against this first. Features that require
  persistent server-side storage (accounts, saved channel lists, chat history) are
  a deliberate exception, not a default ŌĆö flag the tradeoff explicitly rather than
  building it in quietly.
- **No fake controls.** Don't ship a toggle/button that does nothing to match a
  mockup. Where the mockup drew interactive switches for things that are already
  always-true and non-configurable (P2P mesh, in-memory only, blind signalling),
  the app shows read-only info cards instead ŌĆö a fake toggle is actively
  misleading. Same reasoning retired the mockup's "Zoom" button (no action to wire
  it to) and kept it wheel-only.

## Scaling ceiling & regulatory considerations (SFU migration, 2026-07-09)

*Not legal advice ŌĆö this is engineering-team reasoning about the tradeoff, captured so
the SFU decision in `TODO.md` isn't made without this context. Get actual
jurisdiction-specific legal review before committing to an SFU design, especially if
transcoding/mixing/recording is ever added on top of it.*

The mesh architecture's scaling ceiling (see `TODO.md`'s "Scaling ceiling" section) is
real ŌĆö full mesh is N├Ś(N-1) connections, and quality tiers (`_qualityTier()`) only
shrink each connection's bitrate, they don't reduce the connection *count*. The actual
fix at large room sizes is an SFU (Selective Forwarding Unit): one upload per client,
fanned out server-side. That's a deliberate architectural shift away from "no info held
on a server," so before picking it up, it's worth being precise about what legal
exposure it would and wouldn't add ŌĆö a concern raised directly by the user.

- **It's very unlikely to trigger age-verification/ID-check laws** (e.g. Australia's
  social-media minimum-age rules, in force since ~Dec 2025). Those regimes key off
  *product features* ŌĆö persistent accounts, a social graph, content
  discovery/search across users, ongoing feed posting ŌĆö not transport architecture.
  Most explicitly carve out services whose sole/primary purpose is messaging or
  voice/video calling. Zoom and Teams run centralized SFUs and aren't captured by
  these laws; the carve-out is about *what the product is*, not *how media moves*.
  Peek's no-accounts, ephemeral-room design is what keeps it on the "communication
  tool" side of that line, and an SFU migration wouldn't change that classification
  by itself ŌĆö only adding social features (profiles, directories, discovery, feeds)
  would.
- **The real legal risk is different: carriage-vs-hosting classification.** Today's
  TURN relay (fallback only) is a blind byte-forwarder ŌĆö it relays DTLS-SRTP
  encrypted packets without ever holding decryption keys, which is legally similar to
  a telecom carrier ("mere conduit" under the EU's e-Commerce Directive/DSA,
  common-carrier treatment under Australia's Telecommunications Act): no technical
  capability to inspect content, so no reasonable expectation to moderate it. A pure
  forwarding SFU (just picking which simulcast layer to relay per receiver) can
  preserve this property, especially combined with client-side E2EE via WebRTC
  insertable streams, so the server never holds keys. But an SFU that needs to
  **transcode, mix, or compose** streams (audio mixing for large calls, recording,
  server-side composition) typically has to decrypt server-side, even with nothing
  ever written to disk ŌĆö and that's the point regulators/plaintiffs can run a
  "capability implies duty" argument: once you *can* inspect content, even
  transiently, you can be expected to. Australia's Online Safety Act industry codes
  around proactive CSAM detection are the concrete version of this, and they're
  framed around *reasonable technical capability*, not storage.
- **Keep privacy-law exposure and carriage-classification exposure separate** ŌĆö they
  don't move together. Privacy-law exposure (GDPR-style, Australian Privacy Act) is
  about *storage* and stays minimal under a stateless SFU exactly as it does under
  the current mesh, as long as nothing persists server-side. Carriage/hosting
  classification is about *capability to inspect decrypted content*, and only shifts
  if a future SFU design needs to decrypt for transcoding/mixing/recording.
- **If/when the SFU migration is scoped**: favor a pure forwarding design (no
  transcode/mix/record) with client-side E2EE via insertable streams so the server
  never holds keys, and keep logging ephemeral/connection-metadata-only ŌĆö that
  preserves the current legal posture. Recording or server-side composition
  features, if ever wanted, are the things that would actually need a fresh legal
  review, not the SFU migration on its own.

DMs are explicitly out of scope (was a stray future-idea note in the pack, not a real ask). Phases 4-5 were new scope beyond the original pack ŌĆö nothing further currently planned.

Accent tokens are runtime-only, owned entirely by `AccentManager.js` (static token file values are just the pre-JS default/no-FOUC fallback, Violet hue 286).

## Feature narratives moved out of CLAUDE.md's Key Conventions (2026-07-15 housekeeping pass)

These were originally full entries in CLAUDE.md's "Key conventions." Each has been reduced there to a one-line pointer (the still-load-bearing rule, if any) referencing this section ŌĆö moved here verbatim rather than trimmed, since the point of the split is that this file carries the "how we got here" story and CLAUDE.md only carries the standing rule.

- **Draggable self-view PiPs (2026-07-07)**: `#self-view`/`#self-cam-view` (your own screen/cam preview, `UIController.addStream()`) are drag-to-reposition (Discord-style) via `UIController._makeDraggable(el, storageKey)`, using Pointer Events (`pointerdown`/`pointermove`/`pointerup` + `setPointerCapture`) so mouse and touch share one code path. Position is a personal layout preference, not session state, so it's persisted to `localStorage['pipPosition:self-view']`/`['pipPosition:self-cam-view']` and restored on the next stream/reload ŌĆö unlike `peerVolumes`, there's no peerId-reassignment problem here since the storage key is the PiP's fixed role (`self-view`/`self-cam-view`), not a peerId. A dragged element gets `dataset.dragged = 'true'`, which `_updateSelfViewPositions()` (the existing right-edge auto-stacking for when both screen+cam PiPs are visible at once) checks and skips ŌĆö otherwise every stream add/remove would snap a manually-placed PiP back to the corner. `window.resize` reclamps any dragged PiP back on-screen (`_reclampDraggedPip`) in case a saved position from a larger window is now off-screen.
- **Top-bar mesh signal icon (2026-07-08)**: the peer-count pill's old "Xms ┬Ę mesh" text is now the same colored signal-bars icon used per-participant in the members list (`UIController._signalBarsSvg`/`.participant-signal-icon`) ŌĆö `UIController.updateMeshSignal(ms, tier)` (renamed from `updateMeshLatency`, `#topbar-signal-wrap`/`#topbar-signal-icon`), with the exact ms + quality label only as a hover tooltip. `tier` is the *worst* per-peer connection quality across the mesh (`PeerManager._pollConnectionStats`'s `_qualityRank` ranking), not an average ŌĆö one bad connection among several good ones should still show as bad, not get diluted out.
- **Grid/focus tile overlay cleanup (2026-07-08)**: the grid tile's "Stop watching" control used to be a text pill at `top-right`, directly on top of the screen-share resolution badge (`.grid-tile-badge`, also `top-right`) ŌĆö the two visibly overlapped. Fixed by giving each corner exactly one thing: `.grid-tile-stop-btn` (icon-only, red `close` glyph, matches `--red`/hover-fills-red like `.participant-mod-kick`) owns top-right, `.grid-tile-badge` moved to bottom-right, `.grid-tile-name` (mic icon + nickname) stays bottom-left but now has the same dark pill background as `.focus-quality-caption` instead of a bare text-shadow, so both views read as one consistent label style. Focus view previously showed the peer's name **twice** ŌĆö an accent-colored `.focus-sharing-pill` banner top-left ("ŌŚÅ Name is sharing") *and* a separate `.focus-quality-caption` pill bottom-left ("Name ┬Ę WxH") ŌĆö collapsed into one bottom-left pill (`#focus-quality-caption` now holds both the live dot and `#focus-caption-text`); `_updateFocusMeta()` targets `#focus-caption-text` instead of setting `caption.textContent` directly (that would have wiped out the dot span). Focus view also gained the same stop-watching affordance grid tiles already had ŌĆö `#focus-stop-button`, grouped with the existing PiP button in `#focus-icon-btn-group` (`.focus-icon-btn`s are no longer individually `position:absolute`; the group wrapper owns that now, so multiple icon buttons stack horizontally instead of colliding). Clicking it calls `_unwatchTile(focusedPeerId)` **without** clearing `focusedPeerId` or falling through to `updateLayout()`'s auto-refocus branch ŌĆö in a 2-person room `remoteStreams[0]` would still resolve to the same peer, so an unconditional refocus would instantly re-watch the very stream you just paused. Instead `_renderFocusWatchState()` toggles a dedicated `#focus-paused-overlay` (reuses the exact grid-tile paused-overlay classes/copy ŌĆö "Click to watch ┬Ę stream paused ┬Ę saving bandwidth" ŌĆö for one consistent paused look everywhere) over the video, hiding the caption/icon-buttons until the overlay itself is clicked to resume (`_watchTile` + restore `srcObject`). Called from `focusStream()`, the stop/resume handlers, and every `updateLayout()` focus-branch pass, so it can't drift out of sync with `watchedTiles`.
