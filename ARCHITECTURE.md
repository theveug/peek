# Peek – Architecture & Design History

Human-facing design rationale and the full history of the visual redesign. This
file is **not** auto-loaded into an agent's context every session (unlike
`CLAUDE.md`) — it's the "how we got here" record, read on demand when a task
actually needs the backstory. The still-active rules and gotchas that must be
respected on *every* change live in `CLAUDE.md`'s "Key conventions"; this file is
the narrative behind them.

See also:
- `CLAUDE.md` — agent operating rules, file map, context-budget rules, active invariants/gotchas.
- `TODO.md` — active backlog only.
- `CHANGELOG.md` — completed work, most-recent first.

---

## Design principles

- **No info held on a server.** No accounts, no message/chat history persistence,
  no persisted room list. Sessions are in-memory only and disappear the moment the
  last peer leaves (`SessionManager.js`). Media is P2P wherever possible; even when
  a TURN relay is needed as fallback, it's blind-relaying encrypted bytes, not
  storing or inspecting anything.
- When weighing a new feature, check it against this first. Features that require
  persistent server-side storage (accounts, saved channel lists, chat history) are
  a deliberate exception, not a default — flag the tradeoff explicitly rather than
  building it in quietly.
- **No fake controls.** Don't ship a toggle/button that does nothing to match a
  mockup. Where the mockup drew interactive switches for things that are already
  always-true and non-configurable (P2P mesh, in-memory only, blind signalling),
  the app shows read-only info cards instead — a fake toggle is actively
  misleading. Same reasoning retired the mockup's "Zoom" button (no action to wire
  it to) and kept it wheel-only.

## Visual redesign (Phases 0-5 — all complete)

Based on a Claude Design handoff pack (`peek-design.zip`, originally from Downloads — `README.md` + `Peek.dc.html` prototype; the prototype is inline-styled preview-runtime HTML, **not** meant to be copied verbatim, just a spec for appearance/behavior). Phased: 0 decisions → **1 tokens/fonts (done)** → **2 screen re-skins — all sub-phases done (2a lobby, 2b top bar/Invite popover, 2c members sidebar, 2d stage/controls dock, 2e chat/files/polls panel)** → **3 settings panel (done)** → **4 removed the legacy glass/gradient background (done)** → **5 Background tint picker (done)**. DMs are explicitly out of scope (was a stray future-idea note in the pack, not a real ask). Phases 4-5 were new scope beyond the original pack — nothing further currently planned.

- **Phase 1 tokens**: `tailwind.css`'s `:root`/`body.light` now have a second, *additive* token set (`--bg1`..`--bg5`, `--border`, `--input`, `--t1`..`--t3`, `--accent`/`--accentH`/`--accentSoft`/`--accentSoft2`/`--accentText`, `--green`/`--amber`/`--red`) alongside the original `--color-*`/`--surface-*` tokens — original markup keeps working unchanged; only new surfaces (currently just the settings overlay) are built on the new set. Accent tokens are runtime-only, owned entirely by `AccentManager.js` (static token file values are just the pre-JS default/no-FOUC fallback, Violet hue 286).
- **Fonts**: self-hosted Public Sans + Fira Code (single variable-weight `.woff2` each, covers all needed weights) + Material Symbols Rounded (`public/assets/fonts.css`, files in `public/assets/fonts/`). `html, body` now default to Public Sans; `.font-mono-app` utility class and `.chat-markdown code` use Fira Code — **most of the app still hasn't been swept to actually use `.font-mono-app`** for room codes/latency/etc. Material Symbols is the *unsubsetted* full icon set (~5.3 MB, one-time cached download) — subsetting to just the ~30 icons actually used is a worthwhile follow-up, not done yet.
- **Phase 3 settings panel**: `#settings-modal` went from a small centered modal (the original bug report: too many fields, no scroll region, clipped on short viewports) to a full-frame overlay — 262px left nav (5 sections) + a content column with its own `overflow-y-auto`. Owned by the new `public/client/SettingsPanel.js`, same `#settings-modal` id/open-close contract so nothing else had to change. **Every field applies + persists immediately on change now — there's no Save/Cancel step anymore** (matches the design spec's interaction model; deliberate behavior change from the old form-submit pattern). Nickname change fires on `change` (blur), not `input`, so it doesn't broadcast a nickname update per keystroke.
  - **One settings surface everywhere**: the standalone `/settings` page (`settings.html`/`settings.js`) was retired — it was already dead (nothing linked to it) and out of sync (missing several fields). `lobby.html` now carries its own copy of the settings-overlay markup + a gear button, and instantiates `SettingsPanel` with no `ui`/`peerManager` (both optional in the constructor) so pre-room settings still load/persist to `localStorage` without erroring on the parts that need a live connection (status picker, live quality re-apply, nickname broadcast — all guarded with `?.`).
  - **Privacy & P2P section deliberately deviates from the mockup**: it shows read-only info cards (not interactive toggles) for things that are already always-true and non-configurable (P2P mesh, in-memory only, signalling server blind to content) — the mockup draws these as toggle switches, but shipping fake toggles that do nothing would be actively misleading. The one *real* toggle there is `autoAcceptFiles`.
  - **Gotcha already hit**: the custom `.settings-switch` toggle (checkbox + a `.settings-switch-track` sibling span for the visual pill/thumb) needs `pointer-events: none` on the track — without it, the visually-on-top track span intercepts every click meant for the checkbox underneath, so the toggle looks clickable but silently does nothing. Caught by `tests/file-consent.mjs` failing to check the box, not by eyeballing it.
- **Quick room settings popover (2026-07-04, follow-up to Phase 3)**: not every setting deserves a trip through the 5-section full panel — status, `autoAcceptFiles`, `followActiveSpeaker`, and screen/cam quality are things people plausibly want to flip mid-room, so they also live in a lightweight popover off `#quick-settings-button` (`public/client/QuickRoomSettings.js`). It's a second UI surface only — same `localStorage` keys and `PeerManager` methods as `SettingsPanel.js`, no duplicated data/validation logic, and its "All settings" link hands off to the real `SettingsPanel.open()`. The status control changed shape here too: the old inline 3-button segmented picker (still what `SettingsPanel.js`'s Profile section shows) vs. the popover's `#quick-status-trigger`/`#quick-status-menu` click-to-open dropdown — same `peerManager.setManualStatus()` underneath either way. Screen/cam quality in the popover only exposes resolution (not fps) to keep it compact.
- **Phase 2a lobby re-skin (2026-07-04)**: `public/lobby.html` rebuilt from a small `max-w-md` card onto Phase 1 tokens — a 900px-wide column: hero (squircle accent icon + "Start a room" + subcopy, blurred `--accentSoft` radial glow behind it via `.lobby-glow`), a 2-col `.lobby-card` grid (Create/Join), the Saved Rooms list re-skinned into a `.lobby-recent-grid` of cards, and a static 4-item privacy footer. New CSS lives in tailwind.css's `/* ===== Lobby (redesign, Phase 2a) ===== */` section.
  - **Stepper replaces the max-participants `<select>`**: `#create-max-peers` is now a readonly `<input type="number">` (min 2, max 12, default 6) driven by `#create-max-peers-dec`/`-inc` buttons — same id/value contract as before, so `#create-form`'s submit handler didn't need to change.
  - **Join input gained paste-link support**: a `paste` handler takes the pasted text, strips non-alphanumeric, and keeps the trailing 5 characters — so pasting a full invite link (`https://peek.app/r/AB12C`) extracts the room code instead of concatenating the whole URL.
  - **Recent room cards**: `savedRooms.js`'s `saveRoom()` stamps a `savedAt: Date.now()` on every upsert; `lobby.html` calls `saveRoom()` again on a successful "Rejoin" click so lastActive stays current. Each card's avatar hue is computed client-side from the room's code+label via a string hash (`hueFromString()`) — deterministic per room, nothing persisted.
- **Phase 2b top bar + Invite popover (2026-07-04)**: the room page had **no persistent top chrome at all** before this — just a floating `#room-header` pill plus the hover-reveal bottom `#controls` bar. User chose the mockup's literal always-visible 58px bar over an enhanced floating-pill alternative. New `.topbar` row inserted as the first child of the root `flex flex-col h-screen` wrapper (`public/index.html`), so the existing `flex-grow` row below it just shrinks via flexbox.
  - **Left**: brand + room-state pill (`#topbar-room-name`/`#topbar-room-lock`/`#topbar-room-code`, populated via `UIController.setRoomMeta()` from `App.js`'s `'init'` handler). **Right**: peer-count+latency pill (`#topbar-peer-count`/`-cap`, `#topbar-latency`), the Invite popover, the **relocated** `#settings-button`/`#quick-settings-button` (moved out of the bottom controls dock — pure DOM relocation, same ids), and an identity pill (`#topbar-identity-*`).
  - **`#share-button`'s old silent instant-clipboard-copy behavior was removed entirely** — replaced by `InvitePopover.js`, which shows the room code and a direct link with their own Copy buttons and a toast for feedback.
  - **Mesh latency is a new data point**: `PeerManager._pollConnectionStats()` already fetched per-peer RTT every 3s but discarded the raw ms; now it accumulates valid `rtt * 1000` samples and calls `UIController.updateMeshLatency(avgMs)` once per poll tick — reuses the existing timer.
  - **Server plumbing**: `WebSocketServer.js`'s `'init'` payload now includes `hasPassword`/`maxPeers` (already returned by `getSessionMeta()`) so the client can render the lock icon and cap without a new endpoint.
  - **Cascade-layers gotcha (origin of the standing rule in CLAUDE.md)**: toggling a `.material-symbols-rounded` icon's visibility via the Tailwind `.hidden` utility silently didn't work — Tailwind v4 wraps utilities in CSS `@layer`s, and any unlayered rule beats any layered rule regardless of specificity/order; `fonts.css`'s `.material-symbols-rounded` is unlayered, so it always won over `.hidden`'s layered `display: none`. Fixed by toggling `element.style.display` directly.
- **Phase 2c members sidebar (2026-07-04)**: `#members-sidebar`/`_createParticipantCard()` re-skinned onto Phase 1 tokens — square (11px radius) avatars with a deterministic per-peer hue (`UIController._hueFromString()`, same algorithm as `lobby.html`'s but duplicated locally since the two files are separate unbundled entry points), a compact right-side column (signal bars over mic icon, reusing existing `_signalBarsSvg()`/`_micOnSvg()`/`_micOffSvg()`), and a static "Fully meshed" trust card.
  - **Self card moved out of the visible list into a new footer strip** (avatar/nickname/status dropdown/settings gear) — explicit user choice. **The self card is still created in the DOM** (`addSelf()` unchanged) so the existing peerId-keyed update methods keep working unmodified — only visually hidden via `.members-panel .participant-card[data-self="1"] { display: none; }`. Removing it from the DOM entirely would have broken those methods' `!el → addParticipant(peerId)` fallback paths.
  - **Member rows have no hover/click affordance** — the mockup's rows open a private DM, but DMs are out of scope, so `.members-panel .participant-card:hover { background: transparent; }` overrides the shared hover rule.
  - **New `public/client/SidebarIdentity.js`** wires the footer's status dropdown + settings gear — mirrors `QuickRoomSettings.js`'s pattern under new ids so it doesn't collide with the top bar's identical-shaped dropdown. Deliberate second functional entry point, not a redundant copy — the top bar's identity display is glance-only, the sidebar footer adds real click-to-change-status.
  - **Mobile gotcha (origin of the standing rule in CLAUDE.md)**: the footer strip rendered underneath the fixed bottom `#controls` bar (`z-index: 90` vs the drawer's `80`). Fixed with `padding-bottom: calc(11px + 64px + env(safe-area-inset-bottom))` on `.sidebar-identity` in the `@media (max-width: 767px)` block, not a z-index change.
- **Phase 2d stage + controls dock (2026-07-04)**: re-skinned the grid/focus stage and the bottom controls dock. Scoped down via two explicit user decisions: no focus-view filmstrip (focus view is single-stream only today) and no "React" floating-emoji button (doesn't exist).
  - **New stage header row** (`#stage-header`, first child of `#videos`): a Grid/Focus segmented control (`#stage-view-grid`/`#stage-view-focus`, calling `ui.setViewMode()` directly), an active-stream count (`watchedTiles.size / maxWatchedTiles`), and a quality readout that only appears while screen-sharing (`PeerManager.getScreenQualityLabel()`, pushed via `ui.updateStageQuality()` from `startSharing()`/`stopSharing()`).
  - **Grid tiles** (`_buildGridCell()`): rounded corners, a mic icon in the name label (new `this._micEnabled` map keyed by real peerId, updated in `updateParticipantMic()`'s existing choke point), a screen-share resolution badge from the `<video>`'s own `videoWidth`/`videoHeight` on `loadedmetadata`, and the speaking ring (`setSpeaking()` now toggles `.status-talking` on any grid tile matching `peerId` or `peerId-cam`, not just the participant-list avatar).
  - **Focus view**: a "`{name}` is sharing" pill and quality caption, populated in `focusStream()`/`_updateFocusMeta()` from existing data. Zoom/pan stays wheel-only.
  - **Controls dock**: circular pill buttons (`.dock-btn`, 48px); the Leave button becomes a wide labeled red pill. **`#grid-toggle`/`#grid-button` removed** (superseded by the stage-header toggle). **`#theme-toggle` relocated into the top bar** (`.topbar-icon-btn`, same id/handler). A mic-mode badge (`#mic-mode-badge`, "PTT"/"PTM") surfaces `localStorage.micMode` via `ui.updateMicModeBadge()`.
  - **Cascade-layers gotcha hit again**: `updateShareButton()` toggled `#stop-share-button` via `.hidden`, which broke once its markup switched to the custom unlayered `.dock-btn` class (which sets `display: flex`). Fixed with inline `style.display` — this is the second instance that established the standing rule now in CLAUDE.md.
- **Phase 2e chat/files/polls panel (2026-07-04) — Phase 2 complete.** Nothing here was a new feature — reactions, replies, code-block copy, file cards, and poll voting bars already existed; this was a `bg-indigo-*`/`border-indigo-500` → design-token sweep plus a few component restyles. **Polls stay an in-chat card + modal**, not promoted to a third tab like the mockup (an IA change, not a re-skin).
  - **Tab bar** (`#tab-chat`/`#tab-files`): old underline tabs replaced with a pill-segmented control (`.panel-tabs`/`.panel-tab-btn.active`). `UIController._switchTab()` simplified from 8 conditional class toggles to one `.active` toggle each.
  - **Chat composer** (`#chat-input`): rebuilt into a rounded pill (`.chat-composer`) with `--bg5` circular icon buttons. **Added a dedicated send button** (`#send-message-btn`, wired to the same `sendMessage()` Enter already called) — the app previously had no visible send affordance at all, a real mobile-usability gap.
  - **Files tab**: added a persistent "Click to send a file" dropzone card (`#files-dropzone-card`) wired to the existing hidden `#file-input`. Note a pre-existing gap flagged here: drag-and-drop and paste are wired at the Chat tab's DOM level and don't fire while the Files tab is showing. Picking a file from this card calls `ui._switchTab('chat')` so the user sees what they staged.
  - **Chat avatars**: `.chat-avatar` `border-radius` changed from circle to `6px`. The per-name color algorithm (`ChatUI._colorForName()`) was left unchanged.
  - **Poll modal + option bars**: `focus:ring-indigo-500/50` replaced with a `.poll-input:focus` rule; option bars swapped hardcoded indigo utilities for `--accentSoft`/`--accent` via `.poll-option-voted`/`.poll-option-fill`/`.poll-option-hoverable`.
- **Phase 4 — removed the legacy glass/gradient background (2026-07-05).** With Phase 2 done, the chat panel, members sidebar, controls dock, panel-toggle tabs, and update banner were still using the pre-redesign glass-morphism look — `backdrop-filter: blur()` over the animated gradient canvas (`GradientBackground.js` + `#gradient-canvas`). Confirmed via AskUserQuestion: remove entirely to match the mockup's flat solid panels.
  - **Deleted `public/client/GradientBackground.js`** and its two call sites (`App.js`, `lobby.html`) plus the `<canvas id="gradient-canvas">` from both pages.
  - **Flattened every glass surface** onto solid tokens: `.chat-panel`/`.members-panel` → `background: var(--bg2)`; `.dock-bar`/`.panel-tab` → `background: var(--bg3); border: 1px solid var(--border)`; update banner likewise. Removed the dead `.surface-glass`/`.controls-glass` classes, `--surface-glass`/`--controls-glass`/`--controls-border`/`--gradient-1..4` tokens, the `#gradient-canvas` rule, and two orphaned selectors.
  - **Real bug found during verification**: the canvas had quietly been the page's *only* source of a background color (`body`/`html`/`.lobby-page` never declared one), so removing it left plain white showing through the lobby and stage. Fixed with `background: var(--bg1)` on the base `html, body` rule. Any future removal of a full-page background element should check whether anything else declares a base page color.
- **Phase 5 — Background tint picker (2026-07-05), new scope from an earlier "Mesh" design prototype (pre-dates the repo, shown via screenshot).** Adds a 6-swatch "Background Tint" picker to Settings → Appearance (Violet/Slate/Plum/Forest/Sand/Ocean), between Accent Color and Preview — shifts the neutral `--bg1..5`/`--border`/`--input` scale's hue independently of the accent. Built as a near-exact structural copy of `AccentManager.js` (`public/client/BackgroundManager.js`) so a user follow-up — "how easy to add more presets myself?" — has a good answer: one line in a `{ name: hue }` object, no CSS/rebuild.
  - **Real bug found during verification (origin of the standing rule in CLAUDE.md)**: the first implementation applied the override on `documentElement.style` (matching `AccentManager.js`) and silently didn't work while light theme was active — `--bg1..5`/`--border`/`--input` are the first redesign tokens redeclared per-theme (`body.light`), unlike accent tokens which only live in `:root`, and a directly-matched declaration on `<body>` beats an inherited value from `<html>`. Fixed by overriding on `document.body.style` instead.
