## Security

- **Security/privacy pass #3 — offline/self-host (2026-07-07, Fable final review)** — closed every open security item except the documented saved-code-squatting acceptance, and made the app fully functional on a disconnected LAN (zero external requests). (1) **All CDN dependencies self-hosted** — the 44 unpinned/no-SRI jsdelivr tags in `index.html` (incl. DOMPurify, our own XSS sanitizer — the "highest-value open item" carried across two reviews) replaced with pinned local copies in `public/assets/vendor/` (marked@13.0.3, dompurify@3.1.6, jszip@3.10.1, highlight.js@11.9.0 common bundle + autohotkey/powershell/dockerfile); the ~39 individual hljs language packs (an alphabetical paste stopping at "css") were dropped — the common bundle covers more real languages with 3 files instead of 40. The old inline `marked.setOptions` passed a `highlight` option removed from marked in v5 (dead code — real highlighting is `ChatUI`'s post-sanitize `hljs.highlightElement`); the new `markdown-setup.js` keeps only `breaks: true`. (2) **Strict CSP + security headers** on every response (`server.js`): `script-src 'self'` (no `unsafe-inline` — index.html's marked-glue and lobby.html's whole inline module were externalized to `markdown-setup.js`/`lobby.js` to make that possible), plus `frame-ancestors 'none'`, `nosniff`, `Referrer-Policy: no-referrer`, `Permissions-Policy`. (3) **Google STUN hardcode removed** (server `generateIceServers` + the client fallback in `PeerManager`) — every participant's IP pinged Google on each call; now `STUN_URL` env opt-in, empty by default (LAN host candidates need nothing; TURN allocations already yield the srflx candidate). (4) **`utils/Debug.js` no longer hardcodes `NODE_ENV='development'`** — server logging of room codes/names is now opt-in (`DEBUG=1`), so captured stdout on a deployment doesn't accumulate call metadata. (5) **`TRUST_PROXY` env opt-in** wired for both Express (`trust proxy`) and the WS per-IP throttle (X-Forwarded-For) so reverse-proxy deployments get real per-client rate limiting. (6) Vestigial `text`/`messageId` WS destructure removed. Guarded by the new **`tests/offline-selfhost.mjs`** (`npm run test:offline`): blocks every non-localhost request at the browser level, asserts the lobby + room + full markdown/highlight chain work anyway, asserts the CSP header, zero CSP violations, and that `init`'s `iceServers` contains nothing beyond env-configured entries. Full suite (`test:ui`, `test:file-consent`, `test:version-banner`, `test:password-leak`, `test:active-speaker`, `test:mic-gate`) re-run clean.
- **Security hardening pass #2 (2026-07-05, Fable review)** — patched five findings server-side + one client-side, all verified: (1) **WS payload cap** — `new WebSocketServer({ maxPayload: 64*1024 })` in `server.js`; the default 100 MiB let one peer send a huge `nickname-update`/etc. that the server fans out to every other peer (broadcast amplification DoS). (2) **Per-IP failed-join throttle** — the old `failedJoins>=5→close()` counter reset on every new socket, so a script could brute-force passwords 5-at-a-time by reconnecting; added a fixed-window per-IP counter (`WebSocketServer.js`, 20/10min) on top, verified live with a raw-`ws` script (20 wrong attempts across fresh sockets rejected, 21st throttled even with the correct password). (3) **Timing-safe password compare** — `SessionManager.validatePassword()` now uses `crypto.timingSafeEqual` (length-mismatch short-circuits, only leaking length). (4) **Same-session target checks** — `offer`/`answer`/`ice-candidate`/`watch-stream`/`unwatch-stream`/`force-stop-stream`/`kick` all resolved `to` through the *global* `peerMap`, so a client could push signalling (or a creator could kick) into *other rooms* (peerIds are global and survive a same-socket room change); every one now bails unless `getSessionId(to) === sender's session`. `promoteModerator()` likewise now requires the target to be in the room. (5) **Poll-vote spoofing** — `PeerManager`'s `poll-vote` handler trusted the sender-supplied `msg.voterId` (any peer could cast/overwrite votes as anyone); now uses the connection's real `peerId`, like `reaction` already did. Also (same day, follow-up commit): `_polls` → `Map` + `optionIndex` bounds-check + `CSS.escape()` on peer-controlled `querySelector`s (prototype-pollution/bad-selector hardening); `addChatMessage` takes an explicit `isSelf` flag instead of inferring `sender === 'Me'`; TURN credential TTL cut 24h → 2h. Full `test:ui`/`test:file-consent`/`test:active-speaker`/`test:version-banner` suite re-run clean. Remaining findings still open in `TODO.md`'s Security section (CDN pinning/SRI is the big one).
- **Security hardening pass (2026-07-03, Fable review)** — fixed: (1) SVG/MIME same-origin XSS via file transfer — peer-supplied `fileType` was used as the received Blob's MIME and `image/*` got a `target="_blank"` preview link, so an `image/svg+xml` blob would run scripts in our origin with access to localStorage (saved room passwords); now `PeerManager._safeMimeType()` derives MIME from the extension (raster allowlist, everything else `application/octet-stream`). (2) No receive-side file limits — `file-start` size now validated and chunks byte-counted against it, aborting oversized transfers. (3) Room codes now `crypto.randomInt` instead of `Math.random`. (4) Per-IP rate limiting on `/api/create-room` + `/api/validate-room`, and WS force-close after 5 wrong-password joins. (5) Created-never-joined sessions reaped after 10 min (`sweepEmptySessions`); WS `join` validates the 5-char code format so raw clients can't create arbitrary-key sessions. (6) Lazy-recreated rooms now carry the joiner's password forward instead of coming back passwordless. Also: try/catch around data-channel `JSON.parse`, double-join ghost-peer cleanup, dropped the bogus `http` npm package. Verified live: 6/6 smoke checks.

## Moderation

- **Per-peer local block/mute (2026-07-08)** — real gap: the only way to stop seeing/hearing an unwanted peer used to be a moderator force-stopping their stream or kicking them, and not everyone is a moderator. Added a client-side-only "Block" toggle (`UIController._buildBlockControl()`) on every participant's card, available to anyone regardless of moderator status — a personal filter, not a room-affecting action, so it needs no server involvement and no signaling to the blocked peer. Video is suppressed by reusing the existing bandwidth-pause watch/unwatch system (blocking a watched tile calls the same `_unwatchTile()` that off-screen/LRU-evicted tiles use, so the blocked peer's sender actually stops transmitting — real savings, and indistinguishable to them from ordinary tile eviction). Audio is forced through the existing `_applyAudioVolume()` chokepoint to 0 regardless of the peer's saved volume. Chat/reactions/typing/file-offers/polls from a blocked peer are dropped in `PeerManager.js`'s message switch. State (`blockedPeerIds`) is local-only and session-scoped, same non-persistence rule as `peerVolumes` — cleared on every reconnect since peerIds aren't durable. Deliberate scope cut: doesn't retroactively hide chat history already rendered before the toggle. (See `CLAUDE.md`'s "Per-peer local block/mute" convention block.)
- **Room creator = moderator + stop-a-peer's-stream + kick, extended to multi-moderator (2026-07-05)** — real user need: a friend kept forgetting to stop screen-sharing, wasting their own bandwidth, with no way for anyone else to do anything about it. `/api/create-room` mints a random `creatorToken` (128-bit, `server.js`); the creating browser stores it in `sessionStorage['creatorToken']` and presents it on every `join` including reconnects — `SessionManager.claimModerator()` re-validates the token and reassigns `creatorPeerId` each time, since reconnects get a fresh `peerId`. **Two permission tiers**: `isCreator` (the single token-holder) gates kick + promote/demote; `isModerator` (creator + anyone promoted, a `Set`) gates force-stop-stream — deliberately split so a promoted friend can stop the creator's own forgotten stream, while kick/promote/demote stay creator-only. Promotion is session-lifetime only. **Enforcement is entirely server-side** (`WebSocketServer.js`'s four action cases check against the sender) — verified with raw `ws` clients attempting all four actions unauthorized, including self-promote (rejected). Crown badge shows every current moderator to everyone (transparency); per-card kebab menu (`UIController._buildModeratorMenu()`) rebuilt fresh on each open. Kick reuses the existing socket-close cleanup path. Verified via live 3-peer Playwright passes + raw-WS security checks. (See `CLAUDE.md`'s "Room moderator" convention block for the durable rules.)

## Rooms

- **Saved rooms (client-side)** — "Saved Rooms" panel in `lobby.html`, shown above Create/Join only when at least one is saved. Entries (`{code, label, password}`) live in `localStorage` via `public/client/savedRooms.js` — no server involvement. Clicking a saved room's "Join" calls the existing `/api/validate-room` with the saved password: if the room's gone, `SessionManager` lazily recreates it fresh on WS `join`; if alive and the password matches, one-click join; if the saved password no longer matches, falls back to the manual join form with the code pre-filled. Verified live with Playwright.
- **UUID + short invite code system — already satisfied, no work needed** — `SessionManager.js` already keys sessions directly by the 5-char short code and deletes the session when the last peer leaves, giving the desired "recreate = fresh disconnected session" behavior without a separate UUID layer. The originally-planned UUID indirection would have been unused complexity.

## Bandwidth / streaming

- **Adaptive quality tier downgrade — verified live + two bugs fixed (2026-07-09)** — the 7-8/9-10/11-12 participant quality tiers below had only ever been verified for the cap/rejection path, not exercised live. Turned out not to need "7+ tabs each streaming" as originally scoped: `_qualityTier()` counts `Object.keys(this.peers).length` (negotiated connections), not who's actually sending media, so `tests/quality-tier-verify.mjs` (`npm run test:quality-tiers`) uses one real fake-device webcam tab plus 11 media-less filler tabs that just join — cheap enough to run headless in one automated pass. Running it surfaced two real bugs, both fixed: (1) **join-order undercount** — only the WebRTC *offerer* registered its `RTCPeerConnection` in `this.peers` immediately; the *answerer* side recalculated the room's quality tier before its own connection existed, undercounting by one until the next join/leave happened to correct it (this is what made the downgrade look flaky/off-by-one depending on random peer-ID comparison). Fixed by registering the connection synchronously for both sides in the `peer-joined`/`init` handlers before quality is recalculated; `initiateConnection()` now reuses that entry instead of creating a duplicate. (2) **restore-on-leave never actually restored resolution** — `_resolveQuality()` only ever set `{ max }` for width/height (frameRate was already a bare number, which the constraint algorithm treats as `ideal`); a `max`-only constraint makes the browser satisfy the ceiling but never climb back toward a laxer one once it's raised again, so after a big room shrank back down the "Stream quality restored" message fired but the actual video resolution silently stayed capped. Fixed by adding `ideal` (equal to the already-tier-clamped target) alongside `max`. Verified across repeated full 1→12→1 peer runs with no flakiness; `test:audio-senders`/`test:mic-gate`/`test:active-speaker`/`test:ui` re-run clean.
- **Creator-configurable participant cap + adaptive quality** — room creator picks a max participant count (4/6/8/10/12, default 6) in the lobby, stored per-session in `SessionManager.js` (`maxPeers`, clamped 2-12). Enforced at `/api/validate-room` and in `WebSocketServer.js`'s `join` handler (defense-in-depth); client redirects to `/?full=1` on WS rejection. Rooms above 6 get automatic quality downgrades via `PeerManager._resolveQuality()` (tier-by-tier caps at 7-8/9-10/11-12), recalculated live on every `peer-joined`/`peer-left` so quality restores as people leave.
- **Real transceiver pausing (Phase 2)** — unwatched grid tiles genuinely stop transmitting: `PeerManager.setSenderPaused()` calls `sender.replaceTrack(null)` on the per-connection `RTCRtpSender` (no SDP renegotiation), driven by `watch-stream`/`unwatch-stream` signals. Verified via `getStats()`: unwatched tile = 0 bytes received; resumes within ~1.5s of clicking to watch.
- **Watch-slot cap** — `UIController.maxWatchedTiles` (6) with LRU eviction caps how many streams are actively watched/decoded at once, independent of room size. Eviction shows a toast. Verified with 7 senders: cap holds, correct (least-recently-watched) peer evicted.
- **Stop watching button** — watched grid tiles show a "Stop watching" button routing through the same `unwatch-stream` signal as eviction; cleans up the LRU order entry.
- **Focus-view pause/resume** — `updateLayout()`'s focus-mode branch unwatches every remote stream except the focused one on every layout pass, so focus view also gets real bandwidth savings. Verified with `getStats()`: non-focused peer's inbound video = 0 bytes in focus view.
- **Click-to-watch grid tiles (Phase 1)** — grid tiles beyond the 1st show an avatar/"Click to watch" placeholder; single-remote-stream rooms auto-watch. CPU savings only — superseded by real transceiver pausing above for bandwidth.
- **Separate webcam quality controls** — independent "Webcam Quality" resolution/FPS (default 480p/30fps) alongside screen-share quality. `PeerManager.applyCamQualitySettings()` mirrors `applyQualitySettings()`; `toggleCam()` requests saved constraints instead of unconstrained `{video: true}`.
- **Grid/focus tile overlay cleanup (2026-07-08)** — the grid tile's "Stop watching" text pill and the screen-share resolution badge both sat top-right and visibly overlapped. Redesigned both views to be one-thing-per-corner and more Discord-like: grid tiles now have an icon-only red "stop watching" button (top-right), the resolution badge moved to bottom-right, and the name label got a pill background matching focus view's caption. Focus view used to show the peer's name twice (a banner top-left *and* a caption bottom-left) — collapsed into one pill — and gained its own stop-watching button (grouped with the PiP button) plus a paused/click-to-resume overlay reusing the grid tiles' exact placeholder, since it previously had no way to pause the focused stream at all. (See `CLAUDE.md`'s "Grid/focus tile overlay cleanup" convention block.)
- **Screen share audio** — `getDisplayMedia` requests `audio: true`; receive side disambiguates screen audio from mic audio via a `start-sharing` signal (broadcasts the screen stream id) so `ontrack` routes screen audio to a `peerId-screen`-keyed `<audio>` element. `PeerManager.senders[peerId]` keys changed from `'screen'`/`'cam'` to `'screen-video'`/`'screen-audio'`/`'cam-video'` so screen-audio doesn't overwrite screen-video (which would have broken pause/resume). Fixed a surfaced pre-existing bug: `removeStream()` was killing mic audio on every screen-share stop.

## Audio / voice

- **Mic noise suppression via RNNoise (2026-07-10)** — opt-in (Settings → Audio & Mic → "Noise suppression"), client-side only, mirrors the background-blur pipeline exactly but for audio: `PeerManager` now tracks `_rawMicStream` (the actual `getUserMedia` hardware stream) and `micStream` (what's sent — raw stream unchanged when suppression is off, or an RNNoise-denoised stream when on), so toggling live swaps the processing pipeline (`replaceTrack` on every peer's `mic-audio` sender) without restarting the physical mic. Every existing `micStream` consumer (`_reconcileMicGate`'s voice-activity gate, `_localMicLevel()`'s active-speaker analyser) needed zero changes — they already read `this.micStream` generically and now transparently get the denoised signal, which if anything makes voice-activity detection *more* accurate. Vendored `@jitsi/rnnoise-wasm@0.2.1` (Apache-2.0, `public/assets/vendor/rnnoise/rnnoise-sync.js` — the same base64-inlined-WASM build Jitsi Meet ships, needed because `AudioWorkletGlobalScope`'s `addModule()` doesn't await promises so the WASM must compile synchronously). `public/client/rnnoise-worklet-processor.js` is a vanilla-JS adaptation of Jitsi Meet's own `RnnoiseProcessor.ts`/`NoiseSuppressorWorklet.ts` (also Apache-2.0) — reused their exact circular-buffer math (bridging RNNoise's fixed 480-sample/10ms frames to Web Audio's fixed 128-sample render quantum via a `lcm(128,480)=1920`-sample ring buffer) rather than re-deriving it, since that's the one genuinely fiddly part. `public/client/NoiseSuppressor.js` wraps it in the same `start(rawStream)`/`stop()` shape as `VirtualBackground.js`. Added `worker-src 'self'` to `server.js`'s CSP for the worklet load (explicit rather than relying on its fallback to `script-src`). One detail with no video equivalent: `setNoiseSuppression()` must copy the previous track's `.enabled` (mute) state onto the new track across a live toggle, or toggling suppression while muted would silently un-mute the mic. Guarded by `tests/noise-suppression-lifecycle.mjs` (`npm run test:noise-suppression`, pure logic, no browser) plus one-off scripted Playwright verification (single-page: confirmed `addModule()`/`createMediaStreamDestination()` actually fire and zero console errors; two-peer: confirmed `replaceTrack()` fires on the real `mic-audio` sender and the connection survives toggling on/off live) — not kept as permanent test files since CLAUDE.md already documents that headless Chromium's fake-audio-capture reports zero `audioLevel` regardless of injected signal, so real audio-quality verification needs a human ear on real hardware.
- **"They can't hear me" outbound-mic death fixed (2026-07-08)** — root-caused the long-running intermittent audio loss (own speaking ring correct, nothing reaching peers) to two sender-lifecycle bugs in `PeerManager.js`, not to reconnects/room recreation (that path was already clean — rejoining merely *reset* the accumulated damage, which is why it seemed related). (1) **`removePeerStream()` (the `stop-sharing` handler) stripped our own outbound senders**: it looped `pc.getSenders()` (OUR mic/cam/screen toward the ex-sharer, not their incoming stream) calling `removeTrack()` — silently killing audio toward whoever just stopped sharing, with no renegotiation and no error; the mic gate's 200ms `replaceTrack` could temporarily resurrect it until the next renegotiation locked the transceiver receive-only, then it was dead until rejoin. Now the function only removes the video tile. (2) **`receiveOffer()` detected an existing mic sender via live track kinds** (`s.track && s.track.kind === 'audio'`) — the exact null-track anti-pattern CLAUDE.md warns about: a sender gated closed by `_reconcileMicGate` (muted in any mode, or silent in voice-activity mode — i.e. most of the time) has `.track === null`, so any renegotiation offer arriving during silence re-added the mic, creating a never-negotiated duplicate transceiver *and* overwriting `senders[from]['mic-audio']` with it — the gate then drove a dead sender while the real one stayed stranded at `replaceTrack(null)`. Now checks `senders[from]?.['mic-audio']` like screen/cam always did. Regression-tested in `tests/audio-sender-lifecycle.mjs` (`npm run test:audio-senders`); also fixed `tests/mic-gate-logic.mjs`'s stale manual-mute expectation (failing since the gate-every-mode change) and updated `tests/ui-smoke.mjs` to the 2026-07-08 reopen-only panel-tab behavior (it was asserting the pre-redesign tabs). Full suite re-run clean.

## Rooms / UI

- **Webcam background blur (2026-07-09)** — opt-in Settings toggle (Screen & Video) runs your outgoing webcam through client-side segmentation entirely in the browser before it's ever sent, blurring everything behind you: no coordination with peers, no server involvement. Uses a newly self-hosted `@mediapipe/selfie_segmentation@0.1.1675465747` (Apache-2.0, `public/assets/vendor/mediapipe/`, ~12MB — both SIMD and non-SIMD WASM variants vendored after confirming only one is fetched per session but the SIMD-only set 404'd in this project's test environment) wrapped by a new `VirtualBackground.js`. `PeerManager` now tracks the raw camera stream (`_rawCamStream`) separately from the outgoing stream (`camStream`, either the raw stream unchanged or a canvas-capture composite), so `setBackgroundBlur()` can live-swap the pipeline without restarting the physical camera, and `switchCamera()`/`toggleCam()`/`stopCam()` all correctly manage both streams' independent lifecycles. Required adding `'wasm-unsafe-eval'` to `server.js`'s CSP `script-src` — WASM compilation is blocked outright under a strict CSP without it; this is a narrower CSP Level 3 keyword permitting only WASM compile/instantiate, not general `eval()`, so `'unsafe-eval'` itself remains absent. Verified via a scripted check confirming the outgoing track becomes a genuine canvas-capture stream (opaque browser-generated label, not the raw device's), zero console errors, zero failed vendor-asset requests, and a clean live on/off swap while the camera stays running; full existing test suite (offline/self-host, UI smoke, quality-tiers, audio-sender-lifecycle, mic-gate, active-speaker) re-run clean afterward.
- **@mentions in chat (2026-07-09)** — `@nickname` tokens are detected receive-side in `ChatUI._processMentions()` (matched against every current participant's nickname via a new `getAllNicknames` dependency wired from `UIController._allNicknames()`, longest-name-first so one nickname that's a prefix of another can't steal the match), wrapped in a highlighted `.chat-mention` pill visible to every reader — same choke point (`_finalizeMarkdownBody`) as syntax highlighting and link previews, so it covers both plain chat messages and grouped file-caption messages. Operates on the DOM post-sanitize (walking text nodes, skipping ones inside `<a>`/`<code>`), not the HTML string, so it can't break out of an existing tag/attribute. A message that mentions the local user gets a persistent `.chat-message-mentioned` bubble highlight (stays visible even after it's been read, like Discord) plus, only while the chat panel hasn't been seen yet, a distinct `#mention-indicator` "@" badge — separate from the general `#new-message-indicator` dot — so a mention can't get lost among ordinary unread messages in a busy room.
- **Lobby unreachable below the fold on mobile (2026-07-09 fix)** — the room page's "prevent page scroll on mobile" rule (`@media (max-width: 767px) { html, body { overflow: hidden !important; position: fixed !important; ... } }`, `tailwind.css`) applied globally to `html`/`body`, which both pages share with no page-identifying class (`<body class="dark">` on both). On the lobby, whose content is taller than one mobile viewport, this locked scrolling entirely — mobile users could only ever see whatever fit above the fold (hero + Create room card), with Join room, saved rooms, and the privacy footer all present in the DOM but permanently unreachable. Fixed by scoping the rule to `html:has(#videos)`/`body:has(#videos)` — `#videos` is the room stage container and only exists on the room page. Verified with a scripted mobile-viewport check: reproduced the bug against the pre-fix CSS (`position: fixed`, `canScroll: false`, Join control and footer both below the 700px viewport with no way to reach them), confirmed the fix restores scrolling and both become reachable, and confirmed the room page's own mobile fixed-viewport behavior is unchanged.
- **QR code for room join (2026-07-09)** — the Invite popover (`InvitePopover.js`) now renders a scannable QR code of the direct join link below the room-code/link fields, generated entirely client-side via a newly self-hosted `qrcode-generator@2.0.4` (`public/assets/vendor/qrcode-generator.min.js`, pure JS, no dependencies, MIT) — no server involvement, consistent with the offline-first vendor policy. Rendered as an SVG (`createSvgTag({ scalable: true })`) inside `.invite-qr-code`, which forces a white background regardless of theme since the generator bakes in black-on-white modules and a dark surface behind them would kill scannability.
- **Idle/away auto-detection (2026-07-09)** — status auto-flips to Away after 15 minutes of no mouse/keyboard/touch input (`App.js`'s `resetIdleTimer`), same manual-status-respecting logic the existing tab-visibility auto-away already used. Refactored that shared logic out of `PeerManager.handleTabVisibility` into `_reconcileAwayStatus(away)`, now called by both it and the new `handleIdleChange(idle)` — never overrides a manually-chosen DND, and restores whatever status was manually chosen before (not hardcoded 'online') once activity resumes, so a manually-chosen Away also stays put.
