# Peek – Roadmap / Polishing Tasks

Living backlog for Peek. Check this at the start of a session for pending work; update it when items are completed or new ones come up.

See **Design principles** in `CLAUDE.md` before adding new items — weigh new features against "no info held on a server."

## Scaling ceiling (priority — full mesh, will eventually bite)

- [ ] **SFU migration path (longer-term, conflicts with "no info held on a server")** — the actual fix for large rooms is a Selective Forwarding Unit: clients send one upload to a media server, which fans it out, turning N×(N-1) mesh connections into N connections total. This is the real answer if room sizes need to grow past the mesh-practical limit, but it's a meaningful architecture shift and a media server is exactly the kind of "thing in the middle" the no-server-storage principle was written to avoid — needs a deliberate decision later, not a casual addition.
- [ ] **Note: TURN doesn't raise the mesh ceiling, it lowers it** — TURN relay doesn't reduce the number of times a stream must be uploaded (each peer connection still gets its own relay allocation), it just reroutes each copy through the relay server. That adds a second, shared bottleneck (the relay's own bandwidth/CPU, split across every active relayed room) on top of each client's own capacity, plus an extra network hop. Keep this in mind when picking participant caps — a TURN-heavy deployment should cap lower than a mostly-direct-P2P one, not higher.

## Desktop / Electron

- [ ] **Electron wrapper (own repo)** — Browser tabs can't capture keyboard events when unfocused, so push-to-talk/push-to-mute only work while the tab is active. A generic desktop shell (not tied to one hosted Peek instance) would get `globalShortcut` for system-wide hotkeys, tray icon, and would eliminate Chrome's bulky in-tab "sharing" banner via `desktopCapturer` instead of `getDisplayMedia`. Since Peek is self-hostable per-domain (TURN config is per-deployment via `.env`), the shell just navigates to whatever room URL it's given — `https://john.com/room234`, `https://other.com/room234`, or a LAN address — same as a browser tab; it never bundles or duplicates any frontend code, so it needs its own repo (separate build/release/codesigning pipeline) with zero risk of drifting out of sync with this repo's frontend. LAN/offline-from-internet use isn't extra work — falls out for free from self-hosting + WebRTC ICE finding local candidates without STUN/TURN.

## Collaboration

- [ ] **Rejected: local contacts / friends list** — considered a client-side contacts system (local keypair + `localStorage` label per contact, added via out-of-band invite link, no server directory/discovery). Dropped because without accounts there's no cross-room presence, so "friending" someone only pays off if you can PM them outside a room — and *that* needs presence tracking (server knowing a contact is online right now), which is a bigger step against the no-server-storage principle than a contacts list alone justifies. **Saved rooms** below covers the actual need (reconnecting with the same people) without any of that. Revisit only if async P2P messaging (outside any room) becomes a real goal — see Direct messaging note below.
- [ ] **Direct messaging (P2P data channel)** — private 1-to-1 chat between any two participants via `RTCDataChannel` on the existing peer connection, so messages never touch the signaling server at all. Click a participant's avatar/name to open a scoped DM panel alongside the group chat. Implementation lives in `PeerManager.js` (open a named data channel per peer on connect, handle `ondatachannel` on the receiving side). Works identically in both the hosted web version and the future Electron build since data channels are P2P after ICE — the signaling server is only involved in the initial SDP handshake, not in any message routing. Note: a small hosted signaling server is intentionally kept for that handshake step; STUN (Google's free servers) handles NAT traversal for the media/data connection itself for the vast majority of users, so no port forwarding or TURN relay is needed in practice.
- [ ] **Screen annotation/drawing** — overlay a canvas on the shared screen so participants can circle/mark things, useful for code review.

## Rooms

- [x] **UUID + short invite code system — turned out to already be satisfied, no work needed** — investigated while building Saved Rooms below. `SessionManager.js` already keys sessions directly by the 5-char short code (`generateUniqueShortCode()` in `server.js`) and deletes the session the moment the last peer leaves (`SessionManager.removePeer()`). That already gives the desired behavior — recreating a room with the same code later produces a fresh, disconnected session with no continuity to the original — without needing a separate UUID layer. Left as-is; the originally-planned UUID indirection would have been unused complexity.
- [x] **Saved rooms (client-side)** — "Saved Rooms" panel in `lobby.html`, shown above Create/Join only when at least one is saved. Entries (`{code, label, password}`) live in `localStorage` via `public/client/savedRooms.js` (`getSavedRooms`/`saveRoom`/`removeRoom`) — no server involvement. "Save this room to this device" checkbox + optional label field on both the Create and Join forms. Clicking a saved room's "Join" button calls the existing `/api/validate-room` (same endpoint the manual join form uses) with the saved password: if the room's gone, `SessionManager` lazily recreates it fresh on WS `join` (see UUID item above) so the click just works; if it's still alive and the password matches, one-click join with no re-entry; if the saved password no longer matches (someone else recreated the room with a different password), shows "exists but the saved password is wrong — someone else may have recreated it" and falls back to the manual join form with the code pre-filled. Verified live with Playwright: save-on-create, panel visibility, one-click join (plain and password-protected), and remove all round-tripped correctly against the running server.

## Polish

- [x] **Files tab** — a tab in the chat/sidebar panel listing every file shared during the current session (filename, sender, size, download link). Files are kept as raw Blobs in memory as they arrive so they remain available after the chat message scrolls or the object URL is revoked. A "Download all" button zips them client-side via JSZip and triggers a single bundle download — no server needed. Caveat: only files received during *your* time in the session are available (late joiners miss earlier files); everything disappears when the session ends, which is consistent with Peek's no-persistence design.
- [x] **Connection quality indicator** — signal strength icon per participant, computed from ICE stats (latency / packet loss).
- [x] **Leave Room button** — there was no way back to the lobby short of manually editing the URL. Added a red "Leave Room" button (`#leave-room-button`, `index.html`) to the control bar. Building it surfaced a real pre-existing bug: `App.js`'s `socket.onclose` unconditionally scheduled a reconnect (`setTimeout(connect, 2000)`) with no way to opt out, and the socket was never explicitly closed on navigation — it relied on the browser tearing it down as a page-unload side effect, which isn't always a prompt, clean WS close handshake. Combined, a user could leave and immediately rejoin the same room (e.g. via Saved Rooms' one-click join) and see a stale "ghost" copy of themselves lingering as a second participant until the 30s-ping/60s-terminate reaper on the server eventually reaped it. Fixed by adding a `leavingRoom` flag `App.js` checks in `onclose` before scheduling a reconnect, and having the Leave button call `socket.close()` explicitly (synchronously, before `window.location.href` navigation even starts) instead of depending on unload timing. Verified live: join → leave → immediately rejoin the same code in the same tab now settles at 1 participant, not 2.
- [x] **Fixed duplicate participant cards on reconnect (server restart / network blip)** — App.js's auto-reconnect (`socket.onclose` → `setTimeout(connect, 2000)`, intentionally kept for real network blips — see the `leavingRoom` flag above) gets a brand-new `peerId` from the server on every connection. `UIController.addSelf()`/`addParticipant()` key their DOM cards by `participant-${peerId}`, and the existing 'init'-handler cleanup in `PeerManager.handleSignal()` only closed stale `RTCPeerConnection`s/streams (`this.removePeer()`) — it never removed the old participant *cards*, so every reconnect (e.g. the dev server restarting, or anyone's connection blipping) left a stale card behind and added a new one on top, doubling everyone in the "Online" sidebar. Fixed by a new `UIController.clearAllParticipants()` (empties `#participants`, resets `selfPeerId`) called at the top of the `'init'` handler in `PeerManager.js`, before old peer connections are torn down and new ones/cards are added. Verified live: simulated a real dropped connection (Playwright `context.setOffline()` toggle, not just a code review) and confirmed the participant list settles back to exactly 1 card instead of 2.
- [ ] **Keyboard shortcuts panel** — `?` or `F1` overlay listing available shortcuts.
- [x] **Chat closed-panel notifications** — the unread red-dot/sound only fired on `!document.hasFocus()` (background browser tab) or a mobile-only closed-drawer check, so closing the chat panel via its new tab (desktop, same active tab) produced zero notification for incoming messages/files. Fixed with a shared `ChatUI._isChatViewClosed()` used by both `addChatMessage()` and `addFileMessage()`; reopening the panel (desktop and mobile) now also clears the indicator, which it previously didn't do on desktop.
- [x] **Discord-style file compose (caption + stage before send)** — dropping/pasting/picking a file used to call `sendFileToAll()` instantly. Now `App.js` stages files via `ChatUI.addPendingFiles()`, rendered as removable chips in `#attachment-preview` above the message box; actual sending happens in `sendMessage()`, which sends the typed caption as a normal chat message first, then flushes the staged files.
- [x] **Update-available banner** — server stamps a `BUILD_ID` (auto-generated per process start, no manual version bump) into every WS `init` message; a client that reconnects (network blip or actual redeploy) and sees a different id shows a dismissible "new version available" banner with a Refresh button. Deliberately not a forced reload.
- [x] **File-transfer consent (accept/decline, with a trusted-room bypass)** — sending a file is now offer → accept/decline → stream, not blind broadcast: `PeerManager.sendFileToAll()` waits for each peer's response before sending any bytes, so a declined file never gets streamed. Default is an inline Accept/Decline card per incoming file (`ChatUI.showFileOffer()`); a per-session "Auto-accept files in this room" checkbox in Settings (`autoAcceptFiles` in `localStorage`) skips the prompt for rooms you trust — extension/size validation still applies regardless of that setting. Verified live with three two-peer Playwright cases: default prompt-then-deliver, decline-never-delivers, auto-accept-skips-prompt (`tests/file-consent.mjs`).
- [x] **Active-speaker auto-focus** — opt-in via a "Automatically focus on whoever's speaking" Settings checkbox (`followActiveSpeaker` in `localStorage`, default off). `PeerManager` polls real audio levels every 200ms (remote via `getStats()`'s `audioLevel`, local via a Web Audio `AnalyserNode`), with an adaptive per-peer noise floor (see next item), 500ms hold-time, and 1500ms minimum switch gap to avoid flicker during normal conversation; excludes the local user. `UIController.autoFocusTo()` only acts while already in focus view (never forces a view switch) and backs off once the user manually pins someone by clicking a grid tile, resumable by re-saving the setting. Caught-by-testing bugs in the selection logic fixed before shipping (see `CLAUDE.md`). Verified: a standalone unit test of the hold-time/hysteresis decision logic (`tests/active-speaker-logic.mjs`) and a live 3-peer Playwright smoke test of the full settings→focus-view→polling wiring with no console errors.
- [x] **Discord-style speaking ring on participant avatars** — the existing `.status-talking` pulse-ring CSS (previously wired to mic-enabled/disabled, which only means "capable of sound") is now driven by the same real audio-level polling as auto-focus, via `PeerManager.onSpeakingChange` → `UIController.setSpeaking()`. **Fixed a real bug found via live hardware testing**: the first version used one fixed silence-floor constant for everyone, which read as "always speaking" for any mic/room whose self-noise sat above that constant (fan noise, mic gain, open mic with no push-to-talk — reported as "person A is always talking"). Replaced with an adaptive per-peer noise floor (`_updateSpeakingStates()`) that learns each mic's own ambient level and only counts a clear rise above *that* peer's floor as speech; both the ring and auto-focus now consume this same debounced signal instead of raw levels. Verified via unit test that a constant level does *not* stay flagged as speaking once the floor catches up, and that a quiet mic's modest rise above its own (near-zero) floor still registers — real end-to-end verification (ring actually lighting up from a real mic) needs manual testing on real hardware, since Chromium's fake-audio-capture used for headless testing doesn't populate `audioLevel` for injected test audio (confirmed via raw `getStats()` dump — a test-environment limitation, not a code issue).

## Moderation

- [ ] **Room creator = moderator** — the peer who hits "create room" via `/api/create-room` becomes that session's moderator. Needs a creator token returned from the create-room API and verified against the peer's identity when they join over WebSocket (today `SessionManager.js` has no concept of room ownership — `addPeer` treats every joiner the same). Worth deciding whether moderator status survives a disconnect/reconnect.
- [ ] **Kick/disconnect participants** — new signal type (e.g. `kick`) the server only accepts from the verified moderator; server force-closes that peer's WebSocket and broadcasts a `peer-left`-style event so other clients tear down the connection. Straightforward to add to the `WebSocketServer.js` switch.
- [ ] **Per-peer local block/mute** — client-side-only control on a participant's card that immediately stops decoding their video/audio/screen and hides anything they send, until unblocked. Came up as the better fit for "safety from other peers" versus requiring accept/decline on every A/V stream (see this session's conversation) — unlike files, video/audio/screen-share consent-before-delivery would mean a wall of dialogs just to sit in a normal call, and doesn't add real judgment since you'd be accepting blind anyway. This gives per-peer control without that cost.
- [ ] **Room rules: push-to-talk vs. open mic** — store a mode on the session (`SessionManager.js`), broadcast it to joining peers, client UI adapts (e.g. force push-to-talk binding, hide the open-mic toggle). Important caveat: because Peek is a P2P mesh with no media server in the middle, the server can't actually intercept or block audio packets — enforcement is "social"/UI-level (well-behaved clients respect the rule) rather than a hard security guarantee. A modified client could ignore it. Fine for a moderation feature aimed at well-intentioned rooms, just don't market it as airtight.

## Bigger features (post-Electron)

- [ ] **Recording** — record the session locally.
- [ ] **Multiple rooms/channels (Discord-style)** — note: basic multi-room already works today, `SessionManager.js` keys sessions by room code and anyone can create as many as they want. What's missing for a real "server" feel is a *persistent* list of named channels you can browse and return to — today everything's in-memory (`Map`s in `SessionManager.js`) and disappears on server restart. Would need an actual storage layer (even something lightweight like SQLite) before this is worth building out.

## Completed (recent)

- [x] **Security hardening pass (2026-07-03, Fable review)** — fixed: (1) SVG/MIME same-origin XSS via file transfer — peer-supplied `fileType` was used as the received Blob's MIME and `image/*` got a `target="_blank"` preview link, so an `image/svg+xml` blob would run scripts in our origin with access to localStorage (saved room passwords); now `PeerManager._safeMimeType()` derives MIME from the extension (raster allowlist, everything else `application/octet-stream`). (5) Created-never-joined sessions reaped after 10 min (`sweepEmptySessions`); WS `join` validates the 5-char code format so raw clients can't create arbitrary-key sessions. (6) Lazy-recreated rooms now carry the joiner's password forward instead of coming back passwordless. Also: try/catch around data-channel `JSON.parse`, double-join ghost-peer cleanup, dropped the bogus `http` npm package. Verified live: 6/6 smoke checks (rate limit 429, invalid-room rejection, password carry-forward, brute-force cutoff, correct password joins). Not done (noted for later): helmet/CSP headers, self-hosting `marked`/`DOMPurify` instead of CDN, per-file-ID binary chunk routing (concurrent sends from one peer still interleave).

Verified with `getStats()`: non-focused peer's inbound video = 0 bytes while in focus view; round-tripping focus → grid → restores the correct watched/placeholder state for both peers.
- [x] **Screen share audio** — `getDisplayMedia` now requests `audio: true`. Receive side disambiguates screen audio from mic audio via a new `start-sharing` signal (broadcasts the screen stream's id, mirroring the existing `webcam-start` pattern) so `PeerManager.peerScreenStreamIds` lets `ontrack` route the screen's audio track to a `peerId-screen`-keyed `<audio>` element instead of colliding with the mic's `peerId`-keyed one. Also fixed a real pre-existing bug this surfaced: `UIController.removeStream()` was unconditionally calling `removeAudio(peerId)` (the bare mic key) any time a screen video tile was removed, so stopping a screen share used to silently kill that peer's mic audio too — removed; mic cleanup is already independently handled by `removePeer()` on full disconnect. `PeerManager.senders[peerId]` keys also changed from `'screen'`/`'cam'` to `'screen-video'`/`'screen-audio'`/`'cam-video'` to stop the screen-audio sender from overwriting the screen-video sender reference (which would have silently broken Phase 2 pause/resume for anyone with screen+audio). Verified end-to-end with a synthetic getDisplayMedia stream (canvas video + oscillator audio, since headless Chromium has no real desktop to capture): mic and screen-audio render as separate elements simultaneously, stopping screen share removes only the screen one and leaves mic intact, full peer disconnect cleans up both.
